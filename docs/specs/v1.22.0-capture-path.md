# v1.22.0 — Capture Path

## Overview

Two improvements to how cards and decklists enter Project Brew, shipped as one spec with two sequential implementation phases.

**Phase 1** fixes a bug in the existing URL drag flow: dragged TCGPlayer URLs currently resolve to the default printing rather than the specific variant the URL points to. A TCGPlayer product URL contains a numeric product ID (e.g. `tcgplayer.com/product/618860/...`) that maps 1:1 to a Scryfall card via `/cards/tcgplayer/:id`. Phase 1 is a targeted fix to `scryfall.ts` and the drop handler in `page.tsx`.

**Phase 2** adds text capture: paste or drag highlighted text anywhere into the app. The drop handler and a new global paste listener parse the incoming text as either a single card name or a multi-line decklist, then route it to the appropriate existing flow — FindByNameBar preview for single cards, ImportModal for decklists. No new UI components; new wiring only.

Phase 1 is a prerequisite for Phase 2 (shared URL parsing logic). Implement and verify Phase 1 before starting Phase 2.

---

## Phase 1 — URL Variant Resolution

### Problem

The current drag-from-URL flow resolves all TCGPlayer URLs to the default printing of a card. A URL like `tcgplayer.com/product/618860/magic-final-fantasy-jumbo-cactuar-borderless` contains the TCGPlayer product ID `618860`. Scryfall exposes `/cards/tcgplayer/:id` which returns the exact card printing that ID maps to. We are not using it.

### New Scryfall function

Add `getCardByTcgplayerId(productId: string): Promise<ScryfallCard>` to `scryfall.ts`.

```
GET https://api.scryfall.com/cards/tcgplayer/:productId
```

Standard headers. Returns a single `ScryfallCard`. Throws on non-200.

### URL parsing logic

Extract the TCGPlayer product ID from a dropped URL. The product ID is always a sequence of digits immediately after `/product/` in the path.

Pattern: `/product/(\d+)/`

If a match is found → call `getCardByTcgplayerId(id)` → add to deck directly (same path as current: price rescue, `updateActiveDeck`, toast, `setLastAddedId`).

If no match is found → fall through to the existing name-based resolution. Do not change the existing fallback behavior.

### Files changed

- `scryfall.ts` — add `getCardByTcgplayerId`
- `page.tsx` — update the drop handler's URL parsing to extract product ID before falling through to name-based resolution

### Edge cases

- Product ID present but Scryfall returns 404 → fall through to name-based resolution using the card name from the URL slug if parseable, otherwise show existing "card not found" toast.
- URL is a TCGPlayer set page or search page (no `/product/` segment) → fall through unchanged.
- Non-TCGPlayer URLs (EDHrec, Scryfall direct, Card Kingdom) → fall through unchanged. Do not add parsing for other domains in this phase.

### Out of scope

Parsing Card Kingdom, EDHrec, or Scryfall URLs for variant signals. One domain at a time; TCGPlayer is the active pain point.

## Phase 2 — Text Capture

### Problem

There is no way to get a card or decklist into Project Brew by typing or pasting text directly. The only text path is importing a saved `.txt` file. Users who copy a decklist from EDHrec, MTG Goldfish, or a friend's message have no direct paste target.

### What changes

Two new entry points that feed existing flows:

1. **Global paste** — `Cmd/Ctrl+V` anywhere in the app (when no input is focused) triggers text capture.
2. **Drag highlighted text** — dragging selected text from any webpage onto the DropOverlay triggers text capture.

Both entry points share the same parser and routing logic.

### DropOverlay update

The DropOverlay currently activates on URL drags. It needs to also activate on `text/plain` drag types so it gives users a visible drop target when dragging highlighted text.

Update the `dragenter` handler in `page.tsx` to check `dataTransfer.types` — show DropOverlay if `text/uri-list` OR `text/plain` is present.

The DropOverlay copy does not need to change. It already says something generic about dropping cards.

### Parser

A single `parseDroppedText(text: string)` function, co-located with or added to `useDeckImportExport`. Returns one of:

```ts
{ type: 'single'; name: string }
{ type: 'decklist'; lines: ParsedLine[] }  // ParsedLine: { qty: number; name: string; setCode?: string }
{ type: 'unknown' }
```

**Single card detection:** text is a single non-empty line with no leading digit pattern. Examples:
- `Jumbo Cactuar` → single
- `Jumbo Cactuar (Borderless)` → single; strip parenthetical before lookup, use as display hint only
- `Jumbo Cactuar [FIN]` → single with set code; pass set code through to Scryfall lookup

**Decklist detection:** text contains 2+ lines where at least 2 of the first 5 non-empty lines match the pattern `^\d+\s+.+`. Formats supported (same as `.txt` import):
- `3 Ba Sing Se` — qty + name
- `3 Ba Sing Se [TLA]` — qty + name + set code
- `3 Ba Sing Se [TLA] #42` — qty + name + set code + collector number (collector number ignored, not currently used)
- `// Commander:` and `// Sideboard:` section headers — respected, same as file import

**Unknown:** anything that doesn't match single or decklist. Show toast: "Couldn't read that as a card or decklist."

Qty is respected. `3 Ba Sing Se` adds 3 copies, same as `.txt` import.

### Routing

**Single card → FindByNameBar preview flow:**

Call the same resolution path FindByNameBar uses on selection:
1. `searchCards(!"${name}")` to resolve canonical card
2. `getCardPrintings(name, oracle_id)` to load variants
3. Open the FindByNameBar preview panel pre-populated with this card — user sees art variants, oracle text, add button. They confirm and add, same as if they had typed the name themselves.

If a set code was present in the input, pre-select that printing in the variant strip. If the set code doesn't match any printing, select the default and do not warn — the set code hint is best-effort.

If `searchCards` returns no result → toast: "Couldn't find [name]."

**Decklist → ImportModal flow:**

Feed the parsed lines into `pendingImport` state exactly as `handleImportFile` does. ImportModal opens, user chooses current deck or new deck. This is the existing flow; no changes to ImportModal.

### Global paste listener

Register on `document` in `page.tsx` (or a new `usePasteCapture` hook if cleaner). Active when `document.activeElement` is `body` or `null` — i.e. no input, textarea, or contenteditable is focused. Do not intercept paste inside FindByNameBar, deck name input, or any other text field.

On capture: call `parseDroppedText(e.clipboardData.getText('text/plain'))` → route as above.

### State — no new persistence

Neither the paste nor the drag-text flow introduces new persisted state. The FindByNameBar preview and ImportModal states already exist and handle their own lifecycle.

### Edge cases

- User pastes a TCGPlayer URL as text (not a drag) → `parseDroppedText` sees a URL string, which won't match single-card or decklist patterns → `unknown` → toast. They should drag it instead. Do not add URL-parsing to the text path; keep the two capture modes separate.
- User drags a URL (not text) → existing URL drag flow handles it, text capture is not triggered. `text/uri-list` and `text/plain` are both present in URL drags — check for `text/uri-list` first in the drop handler and take the URL path if present. Text path only runs when `text/plain` is present and `text/uri-list` is absent.
- Paste fires while ImportModal is open → ignore. Check for modal open state before processing.
- Paste fires while FindByNameBar preview is open → ignore. Check for preview open state before processing.
- Very large paste (500+ lines) → parse normally; ImportModal handles large lists already via the existing `.txt` import path.
- Decklist with mixed valid/invalid lines → same tolerance as `.txt` import. Lines that don't parse are skipped silently.

### Out of scope

- URL parsing in the paste path.
- Drag-from-image (TCGPlayer image CDN URLs carry no useful card signal; abandoned).
- Parsing Card Kingdom or EDHrec URLs from dragged text.
- Any UI changes to DropOverlay beyond the `dragenter` type check.

## Unvalidated Decisions

**Parenthetical in single card name** — `Jumbo Cactuar (Borderless)` strips the parenthetical before lookup and treats it as a display hint only. The assumption is Scryfall won't match `(Borderless)` as part of the card name. If Scryfall's fuzzy matching handles it gracefully without stripping, don't strip. Claude Code should verify with a test query before committing to the strip logic.

**Single card → preview vs. direct add** — routing single-card text to the FindByNameBar preview rather than direct-add is a deliberate choice: the user gets to confirm the art variant before it lands in the deck. This is slightly more friction than URL drag (which direct-adds) but appropriate for text input since there's no printing signal in a bare card name. If Thurgood finds the confirm step annoying for single names, this becomes a direct-add with undo toast instead. Flag after first use.

**Set code in single card text** — when a set code is present (`Jumbo Cactuar [FIN]`), pre-selecting that printing in the preview feels right. But if the set code is from a format like MTG Goldfish that uses non-standard codes, the pre-select silently falls back to default. This is acceptable; it's best-effort signal, not a guarantee.
