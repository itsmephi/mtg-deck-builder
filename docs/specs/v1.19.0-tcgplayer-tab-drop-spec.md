# v1.19.0 — TCGPlayer Tab Drop

## Overview
Thurgood (and others) frequently browse TCGPlayer for specific cards and want a near-zero-friction way to add that card to the active deck without copy-pasting a name into search. This feature lets the user drag a Chrome tab (or any link) pointing to a TCGPlayer product page onto the app window. On drop, the app parses the URL, resolves it to a card via Scryfall, and adds it to the active deck. Adds are instant with a 4-second Undo toast. Drag-and-drop is desktop-only; mobile is out of scope.

## Data Model
No new persisted fields. One new in-memory state shape:

**`UndoableToast`** — extends the existing toast with:
- `message: string` — e.g. `"Added Sunfall"`
- `actionLabel: string` — `"Undo"`
- `onAction: () => void` — reverses the add (and deletes auto-created deck if applicable)
- `durationMs: number` — `4000` for this variant; existing plain toasts remain `2000`

No migration. No localStorage changes. No changes to `Deck` or `DeckCard` types.

## Behavior

### Drag-over (window-level)
- A drag originates from outside the app (Chrome tab, link, or any source that carries a URL mime type).
- On `dragenter` at the window, if the drag's `dataTransfer.types` includes `text/uri-list` or `text/plain`, show the **drop overlay**.
- Overlay covers the entire viewport above all other UI, including settings view, modals, sidebar, and workspace.
- Overlay is visually passive to layout — it is `position: fixed`, does not scroll with content, and blocks pointer events only for itself (not app interactions behind it; the drag is the interaction).
- Overlay stays visible until `dragleave` fires on the window or `drop` fires.
- `dragleave` fires on child elements during drag; the window-level handler must use the standard pattern of incrementing/decrementing a drag-depth counter or checking `event.relatedTarget` to avoid flicker.

### Drop
1. On `drop`, prevent default (prevents browser navigating to the URL).
2. Extract URL from `dataTransfer.getData('text/uri-list')`, falling back to `getData('text/plain')`.
3. Validate the URL is a TCGPlayer product page. Pattern: any `tcgplayer.com/product/{id}/...` URL. The slug portion after the product ID is the card identifier.
4. If URL is not TCGPlayer or cannot be parsed, hide overlay and show error toast: `"Card not found on TCGPlayer URL"` (standard 2-second toast).
5. Parse the slug. TCGPlayer slugs take the form `magic-{set-slug}-{card-name-slug}`. Extract:
   - **Set slug** — words between `magic-` and the card name (e.g. `march-of-the-machine`)
   - **Card name slug** — the tail of the slug, which becomes the card name after replacing `-` with spaces (e.g. `sunfall` → `"Sunfall"`)
6. Resolve card via Scryfall using existing `searchCards` with a name search.
7. If the set slug is identifiable and Scryfall returns a printing in that set, use that specific printing. Otherwise use Scryfall's default printing (same as the existing `+ Add to Deck` flow).
8. Apply the existing `$0.00` price rescue: if `prices.usd` is null or `"0.00"`, fetch a priced printing via `order:usd` before adding.
9. Determine target deck:
   - If `activeDeckId` is set → add to that deck.
   - If no active deck (Home screen) → create a new deck named `"{Card Name} Deck"` (e.g. `"Sunfall Deck"`), format `"freeform"`, then set it active, then add.
10. Determine target pool:
   - If `activeDeck` exists and `deckViewMode === "sideboard"` → add to sideboard.
   - Otherwise → add to main deck.
11. If the card is already in the target pool, increment quantity by 1 (same as any existing add path).
12. Show Undo toast: `"Added {Card Name}"` with `Undo` button, 4-second duration.
13. Set `lastAddedId` so Workspace scrolls to and highlights the card (same as existing add flow).

### Undo toast
- Appears bottom-center (same position as existing toast).
- Auto-dismisses after 4 seconds; clicking `Undo` dismisses immediately and reverses.
- **Undo action:**
  - If the add incremented an existing card's quantity → decrement by 1.
  - If the add created a new card entry → remove that entry entirely.
  - If the add also auto-created a new deck on the Home screen → delete that deck AND clear `activeDeckId` (returning user to Home screen).
- After dismiss (auto or manual), the toast disappears. Undo is not re-offered.

### Format-violation cards
The card is added silently even if it violates format rules (copy limit, color identity, etc.). Existing soft warning highlights on the card tile handle visual feedback — no change to warning behavior.

### Commander-eligible cards
The card is added as a regular deck card, even if the deck is Commander format with no commander assigned. The user sets the commander manually via existing controls.

## State Tables

### Drop overlay (window-level)

| State | Appearance | On drag-over | On drop | On drag-leave |
|-------|-----------|--------------|---------|---------------|
| Hidden (default) | Not rendered | Becomes Visible if drag carries URL mime type | N/A | N/A |
| Visible | Full-viewport overlay, semi-transparent app-themed backdrop (`bg-surface-backdrop/70`), centered headline `"Drop TCGPlayer tab to add card"`, secondary line `"Release anywhere in the window"` | Stays visible | Becomes Hidden, runs drop pipeline | Becomes Hidden |

### Undo toast

| State | Appearance | On Undo click | On auto-dismiss |
|-------|-----------|---------------|-----------------|
| Hidden (default) | Not rendered | N/A | N/A |
| Shown | Bottom-center pill: `"Added {Card Name}"` text + `Undo` button inline; 4-second timer running | Runs undo action, becomes Hidden immediately | Becomes Hidden |
| Undone (transient) | Brief 2-second toast: `"Removed {Card Name}"` | N/A | Becomes Hidden |

### Workspace view (active deck present)

| State | Appearance | On drag-over | On drop |
|-------|-----------|--------------|---------|
| Normal (no drag) | Deck grid/list as today | Drop overlay appears on top; workspace remains underneath unchanged | Drop overlay disappears; if add succeeds, scroll+highlight fires on the added card via `lastAddedId` (existing behavior) |

### Home screen (no active deck)

| State | Appearance | On drag-over | On drop |
|-------|-----------|--------------|---------|
| Normal (no drag) | HomeScreen tiles as today | Drop overlay appears on top; home screen remains underneath unchanged | Drop overlay disappears; if add succeeds, a new deck `"{Card Name} Deck"` is created, set active, and workspace loads with the card already added and highlighted |

### Search workspace view

| State | Appearance | On drag-over | On drop |
|-------|-----------|--------------|---------|
| Normal (no drag) | Search grid as today | Drop overlay appears on top; search remains underneath unchanged | Drop overlay disappears; card added to active deck (if any) or auto-creates deck. Search view itself does not change — no switch to workspace. User stays in search. |

### Settings view

| State | Appearance | On drag-over | On drop |
|-------|-----------|--------------|---------|
| Normal (no drag) | Settings tabs as today | Drop overlay appears on top of settings | Drop overlay disappears; card is added in background; settings view stays open (does not auto-close) |

### Card modal open

| State | Appearance | On drag-over | On drop |
|-------|-----------|--------------|---------|
| Modal visible | Card modal on top of workspace/search | Drop overlay appears on top of modal | Drop overlay disappears; card is added to active deck. Modal stays open (does not auto-close). |

## Intersection States

The only new persistent UI state is the drop overlay, which is a window-level modal-layer element. Intersections:

- **Drop overlay + existing modal open (CardModal / SampleHandModal / ImportModal / FormatPicker)** — overlay renders above the modal. Modal is not dismissed by the drag lifecycle. After drop, modal remains open.
- **Drop overlay + settings view** — overlay renders above settings. Settings view is not auto-closed by the drop (consistent with the settings overlay contract — only navigation actions close it).
- **Drop overlay + existing toast (e.g. 2s plain toast)** — overlay is above toast. If the drop fires while a toast is visible, the new Undo toast replaces the existing toast (toast system is single-slot).
- **Undo toast + plain toast** — single-slot toast system: a newer toast replaces the older one regardless of type.
- **Undo toast + another drop** — if user drops a second tab while the first Undo toast is still visible, the first Undo is cancelled (toast is replaced; no undo fires). Second drop's Undo toast takes over. The first add remains in the deck.
- **Drop overlay + disabled controls / format warnings on cards** — overlay blocks visual interaction anyway; no interaction-level intersection.

**State precedence:** drop overlay (z-top during drag) > modals > settings > toasts > workspace. The overlay never blocks a drop from completing, and it is dismissed on drop regardless of what is underneath.

## Visual Consequence Chain

On successful drop + add:
- **Drop overlay** — hidden
- **Active deck** — card added or quantity incremented in deck state
- **Workspace grid/list** — new tile or existing tile's quantity badge updates; Workspace scrolls to and highlights the card via `lastAddedId`
- **WorkspaceToolbar** — card count and total value stats update (via `useDeckStats`)
- **Sidebar Decks tab** — if a new deck was auto-created, it appears in the deck list; if no new deck, no change
- **Toast** — new Undo toast appears bottom-center
- **URL bar / browser state** — no change (no navigation)

On Undo:
- **Active deck** — quantity decremented or card removed entirely; if auto-created deck, that deck is deleted and `activeDeckId` clears
- **Workspace** — if the card was removed, its tile disappears; if deck was deleted, workspace returns to Home screen
- **WorkspaceToolbar / stats** — recomputes
- **Sidebar Decks tab** — auto-created deck disappears if it was deleted
- **Toast** — Undo toast replaced by brief 2s `"Removed {Card Name}"` confirmation

On URL-not-recognized (error path):
- **Drop overlay** — hidden
- **Active deck** — no change
- **Toast** — standard 2s error toast: `"Card not found on TCGPlayer URL"`

## Edge Cases

- **User drags something that is not a URL** (e.g. a file, selected text) — `dataTransfer.types` check during `dragenter` omits it; overlay never shows. If it somehow reaches `drop`, URL extraction returns empty; show error toast.
- **TCGPlayer URL with no card-identifiable slug** (e.g. a category page like `tcgplayer.com/magic`) — slug parse fails; show error toast. Do not attempt Scryfall lookup.
- **Scryfall returns zero results** for the parsed name — show error toast with the card name: `"Card not found: {parsed name}"`. No add.
- **Scryfall network failure** — show error toast: `"Could not reach Scryfall. Try again."`
- **Scryfall returns multiple fuzzy matches** (ambiguous name from slug) — use the first exact-name match; if none, use the top result. Do not open disambiguation UI.
- **TCGPlayer set slug does not match any Scryfall printing** — fall back to default printing; do not error.
- **Card is a double-faced card (DFC)** — TCGPlayer slug typically has the front face name only. Scryfall name search resolves it. Existing DFC handling applies (no spec change).
- **Card is already at its copy limit** (e.g. 4 copies of a non-basic in Standard, or 1 copy of a non-basic in Commander) — add anyway; existing soft-warning behavior on the tile takes over.
- **User drops the tab while the same tab's page is loading elsewhere** — no effect; drop handler only reads `dataTransfer`, not the browser's tab state.
- **Rapid consecutive drops** (two tabs in <4 seconds) — first Undo toast is replaced by second. First add is no longer undoable via toast (user can still remove manually via the quantity stepper).
- **Drop during a long-running Scryfall resolve** — overlay dismisses immediately on drop. No "resolving…" state is shown. If resolve takes >1s, user sees nothing until it completes; failure shows error toast. (Flagged as unvalidated — see below.)
- **User drags a TCGPlayer URL from their own bookmarks bar or another app window** — same flow applies; the source doesn't matter, only the URL.
- **Browser blocks cross-origin drag-data read during `dragover`** — per browser security, some browsers hide URL strings until `drop`. Overlay detection uses mime-type presence (`types` array includes `text/uri-list`), not URL content. Validation happens on drop. Some non-TCGPlayer URL drags will briefly show the overlay, then dismiss with error toast — this is acceptable.
- **Drop on top of a card tile in the workspace** — window-level drop handler catches it; no per-tile handling. Drag-to-replace-printing on a specific tile is explicitly out of scope.
- **User is in sideboard view of a non-Commander deck** — add goes to sideboard, per spec. (Commander has no sideboard; `deckViewMode` is locked to `"main"` for Commander decks, so this case can't occur.)
- **Auto-created deck name collides with an existing deck** — append a counter (e.g. `"Sunfall Deck 2"`) matching existing new-deck naming convention.

## Unvalidated Decisions

1. **Overlay visual during drag** — exact styling (opacity, typography, icon, whether to show a dashed border or just a tinted backdrop). Needs prototype to settle. Defaulting to `bg-surface-backdrop/70` with centered headline, but prototype should validate.
2. **"Resolving…" intermediate state** — currently spec says no intermediate state between drop and toast. If Scryfall + price rescue takes >1s in practice, user may perceive app as frozen. Consider adding a lightweight "Resolving…" toast that gets replaced by the Undo toast on success. Resolve via real-network prototype.
3. **Undo toast exact style** — new toast variant with inline button. Visual design (button color, separator, spacing) not yet specified. Needs a prototype pass or a direct decision from Phi.
4. **Browser drag-data timing** — whether `dataTransfer.types.includes('text/uri-list')` is reliably present during `dragenter` in all Chromium-based browsers when dragging a tab from the tab strip. If not, we may need to show the overlay on ANY drag and validate at drop, trading a small false-positive cost for reliability. Needs a real-drag test.
5. **Drop while CardModal is open** — spec says card is added and modal stays open. Confirm this is the desired behavior vs auto-closing the modal so the user sees the add land in the workspace.

## Out of Scope

- **Paste-URL anywhere (Cmd/Ctrl+V detection)** — natural follow-up; not in v1.19.0.
- **Dedicated "Quick Add from URL" input field** — natural follow-up; not in v1.19.0.
- **EDHRec, Scryfall, Moxfield, MTGGoldfish URL support** — only TCGPlayer at launch.
- **Mobile touch equivalent** — no drag-from-tab on touch; no paste-URL fallback added on mobile.
- **Browser extension / bookmarklet** — separate project.
- **Drag-to-replace-printing on a specific card tile** — window-level drop only; no per-tile drag targets.
- **Auto-set as commander when dropping a commander-eligible card into a Commander deck with no commander** — user sets manually.
- **Blocking adds that violate format rules** — existing silent add + soft warning behavior preserved.
- **Disambiguation UI when Scryfall returns multiple fuzzy matches** — use first exact-name or first result.
- **Undo of an undo (redo)** — not supported.
- **Bulk drop (multiple tabs at once)** — browsers deliver one drop event per drag; no multi-card flow in scope.

**Out-of-scope cross-check:**
- *Paste-URL / Quick Add field* — no visual or behavioral overlap; the drop overlay is triggered by drag events only, not paste events. Safe.
- *EDHRec / other sources* — URL validation rejects non-TCGPlayer URLs explicitly, so a later source addition is additive, not overlapping. Safe.
- *Mobile* — drag events don't fire on touch, so mobile naturally gets nothing. Overlay will never appear on mobile. Safe.
- *Drag-to-replace-printing on a tile* — no per-tile drop targets are introduced; all drops are window-level. Safe, but note: if a future feature adds per-tile drop, it will need to coexist with or override the window-level handler.
- *Auto-set commander* — no commander-setting logic is touched. Safe.
- *Format violation blocking* — add path uses the same `handleAddCard` as existing flow; no new rule enforcement. Safe.
