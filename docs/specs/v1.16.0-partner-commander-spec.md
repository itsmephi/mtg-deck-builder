# v1.16.0 Design Spec — Partner Commander Support

> **Status:** DESIGN APPROVED
> **Date:** 2026-03-27
> **Designed in:** Claude Chat design session
> **Backlog item:** Partner commander support — allow 2 commanders for Partner and Friends Forever pairs; deferred from v1.4.0
> **Depends on:** v1.15.0 shipped (current state)

---

## Theme

Commander decks can have two commanders when both cards have a compatible partner ability. The app detects partner keywords, routes the crown interaction to the right action (replace vs. add partner), validates pairings strictly, and warns on invalid combinations — never blocks. Same "app informs, user decides" philosophy as all existing format enforcement.

## 1. Data Model Changes

### Deck Interface

```ts
export interface Deck {
  id: string;
  name: string;
  cards: DeckCard[];
  sideboard?: DeckCard[];
  format: "freeform" | "standard" | "commander";
  commanderIds?: string[];  // CHANGED — was commanderId?: string
}
```

- Max 2 entries. Empty array or `undefined` = no commanders.
- Each entry references a `DeckCard.id` in the deck's `cards` array.
- Dangling refs (card removed from deck) treated as "no commander" — same as existing behavior.

### ScryfallCard Interface

```ts
export interface ScryfallCard {
  // ... existing fields ...
  keywords?: string[];  // NEW — Scryfall returns this on every card
}
```

Scryfall already includes `keywords` in every card response (e.g., `["Partner", "Flying"]`). The `searchCards` function returns raw Scryfall data, so no fetch changes needed — just the TypeScript type.

## 2. Migration

### localStorage Migration (`useDeckManager.tsx`)

```ts
function migrateDecks(rawDecks: any[]): Deck[] {
  return rawDecks.map((deck: any) => ({
    ...deck,
    format: deck.format ?? "freeform",
    // Migrate commanderId → commanderIds
    commanderIds: deck.commanderIds
      ?? (deck.commanderId ? [deck.commanderId] : undefined),
    cards: deck.cards.map((card: any) => {
      if (card.ownedQty !== undefined) return card;
      return { ...card, ownedQty: card.isOwned ? card.quantity : 0 };
    }),
  }));
}
```

Delete the old `commanderId` key during migration — don't carry both. The spread `...deck` will bring it in, but it's unused after migration; the app only reads `commanderIds`.

### Backup/Restore Compatibility

`replaceAllDecks` already calls `migrateDecks`, so restored backups with the old `commanderId` field migrate cleanly.

## 3. Partner Detection Logic (`formatRules.ts`)

### Types

```ts
export type PartnerType = 'partner' | 'partner-with' | 'friends-forever' | null;
```

### `getPartnerType(card: DeckCard): PartnerType`

Detection priority using the `keywords` array (preferred) with `oracle_text` fallback for cards stored before this release:

1. Check `card.keywords` (if present):
   - Contains `"Friends forever"` → `'friends-forever'`
   - Contains `"Partner"` → check oracle text for `"Partner with "` pattern:
     - Found → `'partner-with'`
     - Not found → `'partner'` (generic)
   - Neither → `null`

2. Fallback when `keywords` is absent (pre-existing cards without backfill):
   - Check `oracle_text` (with `card_faces[0]` fallback, same pattern as `isEligibleCommander`)
   - Contains `"Friends forever"` → `'friends-forever'`
   - Contains `"Partner with "` → `'partner-with'`
   - Contains `"Partner"` (standalone word, not part of "Partner with") → `'partner'`
   - None → `null`

**Important:** The oracle text check for generic "Partner" must exclude cards that only have "Partner with [name]" — those are `'partner-with'`, not generic `'partner'`. A card with generic Partner will have the keyword `"Partner"` in its `keywords` array AND its oracle text will say `"Partner"` without "with" following it. The regex pattern: `/\bPartner\b(?!\s+with\b)/i` on oracle text handles this.

### `getPartnerWithName(card: DeckCard): string | null`

For `'partner-with'` cards only. Extracts the named partner from oracle text.

Pattern: `"Partner with "` followed by the card name, terminated by newline, period, or end of string. Use: `/Partner with ([^\n(.]+)/i` — capture group 1 is the partner name, trimmed.

Returns `null` for non-partner-with cards.

### `canPartnerWith(cardA: DeckCard, cardB: DeckCard): { valid: boolean; warning?: string }`

Validates whether two cards form a legal partner pairing:

| cardA type | cardB type | Result |
|---|---|---|
| `null` | any | `{ valid: false, warning: "Neither commander has a partner ability" }` if both null; `{ valid: false, warning: "[cardA.name] doesn't have a partner ability" }` if only A is null |
| `'partner'` | `'partner'` | `{ valid: true }` |
| `'partner'` | `'friends-forever'` | `{ valid: false, warning: "Partner and Friends Forever are incompatible" }` |
| `'partner'` | `'partner-with'` | `{ valid: false, warning: "[cardB.name] can only partner with [partnerName]" }` — unless cardA is the named partner |
| `'partner-with'` | `'partner-with'` | Valid only if each names the other: `{ valid: true }` or `{ valid: false, warning: "[cardA.name] can only partner with [expectedName]" }` |
| `'friends-forever'` | `'friends-forever'` | `{ valid: true }` |
| `'friends-forever'` | `'partner'` | `{ valid: false, warning: "Partner and Friends Forever are incompatible" }` |
| `'partner-with'` | `'partner'` | Valid if cardB is the card named by cardA's "Partner with" text; otherwise `{ valid: false, warning: "[cardA.name] can only partner with [expectedName]" }` |
| any | `null` | `{ valid: false, warning: "[cardB.name] doesn't have a partner ability" }` |

**Edge case — Partner-with + generic Partner:** A card with "Partner with X" can technically pair with X even if X has generic "Partner" (not "Partner with"). This is valid per the rules. The check: if cardA is `'partner-with'` and `getPartnerWithName(cardA)` matches `cardB.name`, valid regardless of cardB's partner type (as long as cardB is commander-eligible).

### `hasPartnerAbility(card: DeckCard): boolean`

Convenience helper: `getPartnerType(card) !== null`.

## 4. Crown Interaction — State Machine

The crown button behavior is context-sensitive. The **entire decision tree** for what the crown shows and does:

### No Commander Set

| Condition | Crown Label | Action |
|---|---|---|
| Any eligible card | "Set as Commander" | Sets card as `commanderIds[0]` |

### One Commander Set (`commanderIds` has 1 entry)

| Condition | Crown Label | Action |
|---|---|---|
| Hovered card = current commander | "Commander ✓" | Removes commander (clears `commanderIds`) |
| Both existing commander AND hovered card have compatible partner ability (`canPartnerWith` returns any result, valid or not — only needs `hasPartnerAbility` on both) | "Set as Partner" | Adds card as `commanderIds[1]`; runs `canPartnerWith` for validation |
| Hovered card does NOT have partner ability, OR existing commander does NOT have partner ability | "Set as Commander" | Replaces `commanderIds[0]` with new card |

**Clarification on "Set as Partner" trigger:** The crown says "Set as Partner" when BOTH the existing commander AND the hovered card each have some partner ability (any type). The compatibility check (`canPartnerWith`) runs after designation — if the pairing is invalid, the red crown appears. This means a user can pair Partner + Friends Forever; they'll get a warning but won't be blocked.

### Two Commanders Set (`commanderIds` has 2 entries)

| Condition | Crown Label | Action |
|---|---|---|
| Hovered card = commander 1 | "Commander ✓" | Removes commander 1; commander 2 becomes sole commander (`commanderIds[0]`) |
| Hovered card = commander 2 | "Partner ✓" | Removes commander 2; commander 1 stays as sole commander |
| Hovered card has partner ability (and either existing commander has partner ability) | "Set as Partner" | Replaces `commanderIds[1]` with new card; revalidates pairing |
| Hovered card does NOT have partner ability (or neither existing commander has partner ability) | "Set as Commander" | Replaces `commanderIds[0]` with new card; `commanderIds[1]` stays; revalidates pairing |

**Revalidation:** Whenever any commander slot changes, run `canPartnerWith(commanderIds[0], commanderIds[1])` and store the result. This drives the red crown display.

## 5. Red Crown — Invalid Partner Warning

### Grid View (VisualCard.tsx)

When a card is `commanderIds[1]` (the partner) AND `canPartnerWith` returns `{ valid: false }`:

- The crown badge background changes from `bg-yellow-500` to `bg-red-500`
- Crown icon color stays white (same as current yellow crown)
- Tooltip shows the warning message from `canPartnerWith` (e.g., "Partner and Friends Forever are incompatible")

When the card is `commanderIds[1]` and the pairing IS valid:
- Crown badge stays `bg-yellow-500` (normal commander treatment)
- Tooltip: "Partner ✓"

Commander 1 (`commanderIds[0]`) always shows the normal yellow crown, never red. The warning is on the partner (second slot) only — the "newer" designation.

### List View (ListCardTable.tsx)

When a card is `commanderIds[1]` and pairing is invalid:

- Crown icon color changes from `text-yellow-400` to `text-red-400`
- Amber `!` warning badge appears with tooltip text from `canPartnerWith`
- Both visual cues match existing warning patterns in list view

When pairing is valid:
- Crown stays `text-yellow-400` (filled), same as commander 1

### Both Views — Hover Overlay States

The hover overlay crown button for commander 2 with invalid pairing:

- **Grid:** Button background `bg-red-500/20` (instead of `bg-yellow-500/20`), crown icon `text-red-400`, label shows warning text in `text-[9px] text-red-400`
- **List:** Red crown icon inline, same `text-red-400`

## 7. Color Identity — Combined

### When It Activates

Color identity checks run when ALL of these are true:

1. Deck format is `"commander"`
2. `commanderIds` has at least 1 entry
3. At least one commander card has a non-empty `color_identity`

### Combined Identity

The allowed color identity is the **union** of all commanders' `color_identity` arrays:

```ts
function getCombinedCommanderIdentity(deck: Deck): string[] | undefined {
  if (deck.format !== "commander" || !deck.commanderIds?.length) return undefined;
  const identities = deck.commanderIds
    .map(id => deck.cards.find(c => c.id === id))
    .filter(Boolean)
    .flatMap(c => c!.color_identity ?? []);
  const unique = [...new Set(identities)];
  return unique.length > 0 ? unique : undefined;
}
```

Example: Commander 1 is `["W", "U"]`, Commander 2 is `["B", "R"]` → combined identity is `["W", "U", "B", "R"]`. Every card in the deck is checked against this combined identity.

### Search Filter

`buildFilterSyntax` in SearchWorkspace uses the combined identity for the `id<=` clause:

```ts
if (format === "commander" && deck.commanderIds?.length) {
  const commanders = deck.commanderIds
    .map(id => deck.cards.find(c => c.id === id))
    .filter(Boolean);
  const combined = [...new Set(commanders.flatMap(c => c!.color_identity ?? []))];
  if (combined.length) {
    syntax += ` id<=${combined.join("")}`;
  }
}
```

## 9. Import / Export

### Export

When a Commander deck has 2 commanders, emit two `// Commander:` lines:

```
// Format: Commander
// Commander: Thrasios, Triton Hero
// Commander: Tymna the Weaver
1 Thrasios, Triton Hero [C16] #46 [owned:1]
1 Tymna the Weaver [C16] #48 [owned:1]
...
```

Order: `commanderIds[0]` first, `commanderIds[1]` second.

### Import

Collect ALL `// Commander:` lines (not just the first). After fetching cards, match each name:

```ts
const commanderNames: string[] = [];
for (const line of lines) {
  if (line.startsWith("// Commander: "))
    commanderNames.push(line.replace("// Commander: ", "").trim());
}

// After fetching...
const commanderIds = commanderNames
  .map(name => fetchedDeckCards.find(c => c.name === name)?.id)
  .filter(Boolean) as string[];
```

Backward compatible — files with one `// Commander:` line still work (array of 1). Files with no commander line → empty array → no commanders.

## 12. `getCardWarnings` Update

The existing `getCardWarnings(card, format, commanderIdentity)` signature stays the same — `commanderIdentity` is already the resolved color array. The caller (Workspace) changes how it computes that array (Section 7), but the function itself doesn't need to know about partners.

**New warning case:** Add a partner pairing warning to the card if it's `commanderIds[1]` and `canPartnerWith` returns invalid. This warning should be added by the caller (Workspace/VisualCard) since `getCardWarnings` doesn't have access to the other commander — or we add an optional `partnerWarning?: string` parameter that the caller passes through.

Recommended approach: caller passes `partnerWarning` as an additional string to append to the warnings array. Keeps `getCardWarnings` focused on per-card rules.

## 13. Keywords Backfill Strategy

**No bulk backfill.** Existing cards in localStorage won't have `keywords`. The partner detection logic (Section 3) falls back to `oracle_text` parsing when `keywords` is absent. This is reliable for partner detection.

Cards added after this release will have `keywords` populated from Scryfall's response (already returned, just previously untyped).

If we later find oracle text fallback is insufficient for edge cases, we can add a `backfillKeywords` function modeled on the existing `backfillColorIdentity` — but that's not needed for this release.
