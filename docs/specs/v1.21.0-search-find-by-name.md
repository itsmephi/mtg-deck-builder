# v1.21.0 — Search as Find-by-Name

## Overview

Project Brew's search surface was built for discovery — NLP parser, category
chips, sidebar filters, a dedicated search workspace. Thurgood doesn't use it
that way. He discovers cards externally (EDHrec, MTG Goldfish, TCGplayer),
arrives knowing the card name, and wants to bring it into a deck quickly while
choosing the art he wants. This spec rebuilds the search surface as a
find-by-name affordance: type name → autocomplete → inline preview with art
variants → add. It removes the discovery scaffolding that no longer earns its
place and replaces the search-add path through CardModal with an inline
preview that keeps the deck visible during the consider-this-card moment.

The deck-context CardModal (art-swap-after-add) is unchanged in this spec and
will be redesigned in a later phase-1 spec; the inline-preview pattern
established here is the design language that redesign should inherit.

## Data Model

No new fields, no migrations. Add flow continues to write `DeckCard` (with the
selected printing's `id`, `image_uris`, `set`, `prices`) into the active deck
via `updateActiveDeck` — same as today.

Removed localStorage keys (cleaned up on next mount, no migration needed since
their absence is harmless):
- `mtg-search-view-mode` — search view toggle is gone
- `mtg-search-filter-active-{deckId}` — sidebar filter activation per deck is gone
- `mtg-search-sort-direction` — search sort UI is gone
- `mtg-sidebar-filters` — sidebar filter state is gone
- `mtg-sidebar-active-tab` — sidebar tab state is gone (Decks tab is the only tab)

## Behavior

### Placement

The search bar lives at the top of the deck workspace, sticky to the top of
the workspace scroll container. It is part of the deck workspace, not the
sidebar. The sidebar's "Search" tab is removed. The collapsed sidebar rail
loses its search icon.

When no deck is active (HomeScreen), no search bar is shown — search exists
to add cards to a deck, and there is no deck to add to.

### Collapsed state (default)

A single-row strip showing a search icon, an input with placeholder text
"Find a card by name…", and a `/` keyboard hint on the right side of the
input. Total height: 52px including the strip's outer padding.

The `/` key, pressed anywhere in the deck workspace, focuses the input. On
macOS, `Cmd-K` also focuses it. The shortcut hint hides when the input has
focus.

### Typing → autocomplete

As the user types, after 1 character, an autocomplete dropdown appears below
the input. It calls Scryfall's `autocompleteCards` (already in `scryfall.ts`)
debounced at 150ms. Up to 8 results visible; more scroll within a 320px
max-height container.

Each row shows: mana cost pips (left), card name (center), set code (right,
only when ambiguous — see "Ambiguity" below). No price, no type, no rulings.
The autocomplete is lean by design.

Keyboard nav: arrow up/down moves focus, Enter selects, Esc clears the input
and closes the dropdown. Mouse: hover sets focus, click selects.

### Selection → inline preview

Selecting a result (Enter on focused row OR click) replaces the autocomplete
dropdown with an inline preview, expanded below the input row. The strip
expands to full preview height; the deck below pushes down. The deck remains
visible above and below the strip — no overlay, no backdrop, no modal.

The preview shows:

- **Left column (220px wide):** card art at full proportion (aspect-ratio
  488/680), default printing = most recent (newest `released_at`).
- **Right column (flex):** card name (heading), mana cost, type line, oracle
  text in a muted box, meta row (set name · rarity · price · "N printings"),
  and action buttons pinned to the bottom of the column.
- **Bottom row (full width):** "Choose art" label with printings count, and
  a horizontal scrollable strip of every printing as 88×~123px tiles. The
  selected printing has a 2px border in `--border-focus` plus a 2px outer
  ring in the surface color.

Action buttons:
- **+ Add to deck** (primary) — commits the card with the currently selected
  printing. After click: preview collapses to 0 height, input clears, focus
  returns to input, toast confirms add ("Added Lightning Bolt"). Toast uses
  the existing 2s plain variant.
- **Cancel** (secondary) — collapses preview, returns to autocomplete with
  input value preserved. Esc key triggers the same.

### Variant selection

Clicking a variant tile updates the preview-art image, the set name in meta,
and the price (variants of the same card have different prices). It does NOT
re-fetch the autocomplete or close the preview. The user can keep clicking
through variants until they pick one to add. This is the core art-swap-as-
creative-act flow.

### After add

Preview collapses. Input clears. Focus returns to input. Toast appears. The
deck workspace scrolls to highlight the newly added card (existing
`lastAddedId` mechanism in `useDeckManager`).

The user can immediately type another name. The fast loop is: type, pick,
choose art, add, type again. No modal mount/unmount, no view changes.

### Ambiguity (same name, multiple cards)

Some card names appear on multiple distinct cards — usually variants like
"Lightning Bolt" appearing on a regular printing and a "Lightning Bolt //
Lightning Strike" double-faced card. Scryfall's autocomplete returns these
as separate entries. When the autocomplete list contains multiple rows with
identical names, show the set code as a disambiguator on those rows only.
Rows with unique names omit the set code.

## State Tables

### Search bar state table

| State | Appearance | On hover | On click |
|-------|-----------|----------|----------|
| Collapsed, empty | Single row, placeholder text, `/` hint | Cursor changes to text on input | Focuses input |
| Collapsed, focused | Same row, `/` hint hidden, cursor in input | N/A | N/A |
| Typing, no results yet | Same row + autocomplete dropdown showing skeleton (3 grey rows) | N/A | N/A |
| Typing, results | Same row + autocomplete dropdown with up to 8 rows | Hovered row gets `--surface-deep` background | Selecting a row opens preview |
| Typing, no results | Same row + dropdown with single "No card matches" row | N/A | N/A |
| Preview expanded | Strip taller; preview content below input row; autocomplete hidden | N/A on preview area; variants and buttons have own hover states | See preview state table |
| API error | Same row + dropdown with "Could not reach Scryfall — try again" row, no skeleton | N/A | Clicking row retries the search |

### Preview state table

| Element | State | Appearance | On hover | On click |
|---------|-------|-----------|----------|----------|
| Art (left) | Default | Full-size card art, currently selected printing | No hover effect | No action — variant strip handles selection |
| Variant tile | Unselected | 88×~123px, 2px transparent border | 2px `--border-default` border | Updates preview-art, set name, price |
| Variant tile | Selected | 2px `--border-focus` border + 2px outer ring | No additional hover effect | No-op (already selected) |
| + Add to deck button | Default | Primary button | Slightly darker bg | Commits card with selected printing, collapses preview, toast |
| + Add to deck button | Disabled | 50% opacity, cursor not-allowed | No effect | No effect — see "intersection states" for when this fires |
| Cancel button | Default | Secondary button | `--surface-deep` bg | Collapses preview, returns to autocomplete with input preserved |

## Intersection States

The new "preview expanded" state intersects with several existing format-rule
states. Format warnings are shown inline in the preview's meta row, between
the price and the printings count. Each warning is a small pill with a
warning icon and text.

| Existing state | Behavior in preview |
|----------------|---------------------|
| **Color identity violation** (Commander format, card has color outside commander's identity) | Pill: "Outside [W][U] identity" in `--warning` color. + Add to deck button is **enabled** — soft warning, not a wall. After add, the card appears in deck with existing color-identity warning treatment. |
| **Copy limit warning** (already at 4 copies in Standard, or 1 in Commander) | Pill: "Already in deck (4×)" or "Already in deck". + Add to deck button is **enabled** — soft warning. Adding a 5th copy works; existing copy-limit highlight in deck handles the visual flag. |
| **Format illegality** (card not legal in current format) | Pill: "Not legal in Standard". + Add to deck button is **enabled** — soft warning. Existing legality treatment in deck applies after add. |
| **Multiple warnings simultaneously** | Pills stack horizontally in the meta row. Order: legality > color identity > copy limit. If the row would overflow, wraps to a second meta row. |
| **Card has no price data** (`prices.usd` null/0) | Existing price rescue runs at add time, not preview time. Preview meta shows "Price unavailable" until rescue completes (rescue is fast — typically <300ms). |
| **Card is double-faced (DFC)** | Variant tiles show the front face. Preview-art shows front face by default with a small "flip" affordance overlaid bottom-right. Existing `getCardPrintings` DFC handling (front-face-name search) applies. |

Precedence rule: warnings in the preview never disable the Add button. The
deck is the source of truth for hard rules; the preview is where you
*consider* a card. Soft warnings inform; they don't gate. This matches
PRODUCT.md's "Help, don't lecture" principle.

## Visual Consequence Chain

When a variant is selected:
1. `preview-art` background image changes to selected printing's
   `image_uris.normal`.
2. Meta row's set name updates (`set_name`).
3. Meta row's rarity updates (`rarity`).
4. Meta row's price updates (`prices.usd`).
5. Selected variant tile gains 2px focus border + outer ring.
6. Previously selected variant tile loses border.
7. Internal "selected printing id" state updates — this is the id committed
   on Add.

When + Add to deck is clicked:
1. `handleAddCard` fires with the currently selected printing.
2. Price rescue logic in existing add handler still applies if needed.
3. `updateActiveDeck` writes the new `DeckCard`.
4. Preview collapses (height → 52px, content fades out at 150ms).
5. Input clears.
6. Focus returns to input.
7. Toast appears: "Added [card name]".
8. Workspace deck stats update (`useDeckStats` reactively recomputes
   `totalCards`, `totalValue`, etc.).
9. `lastAddedId` is set; deck workspace scrolls to and highlights the new
   card (existing behavior).

When the preview is dismissed via Cancel or Esc:
1. Preview content fades out.
2. Strip collapses to 52px.
3. Input value is preserved.
4. Autocomplete re-opens with results for the preserved query.
5. Focus returns to input.

## Edge Cases

- **No active deck.** Search bar is not rendered — there's no deck to add to.
  Adding-a-card flow starts from creating/opening a deck.
- **Sideboard view active.** Search bar visible. Adding via search adds to
  whichever view is active (`deckViewMode`). The Add button label changes to
  "+ Add to sideboard" in this case to make the destination explicit.
- **Format change while preview is open.** If the user changes the deck's
  format while a preview is open, the preview's warning pills re-evaluate
  immediately (color identity, legality, copy limits all re-check against the
  new format). Selected printing is preserved.
- **Commander not yet set, format is Commander.** Color identity check has no
  basis. Skip the color identity warning entirely until a commander is set.
  Copy limit warning still applies (singleton).
- **Long card names.** Card name in preview-name truncates with ellipsis at 2
  lines maximum. Autocomplete name truncates at 1 line.
- **Long oracle text.** Oracle text box has max-height 120px with vertical
  scroll. No truncation — Thurgood needs to read the whole thing to evaluate.
- **Many printings.** Variants strip scrolls horizontally. No upper bound on
  printings — common cards like Lightning Bolt have 25+. Strip uses
  `scroll-snap-type: x proximity` for clean swipe behavior.
- **Card has zero printings with art** (rare, mostly digital-only or token
  cards). Show single fallback printing with placeholder. "Choose art" label
  hides if there's only one printing.
- **Slow network.** Skeleton rows in autocomplete during fetch. Preview shows
  skeleton during initial printing fetch (typically <500ms).
- **Scryfall down or rate-limited.** Autocomplete shows the error row; the
  row is clickable to retry. Existing `searchCards` 429 handling continues
  to apply.
- **Esc with autocomplete open and input non-empty.** Clears input and closes
  autocomplete (single-press dismissal of the entire search interaction).
- **Esc with preview open.** Returns to autocomplete (Cancel-equivalent), as
  documented above. A second Esc clears the input and closes everything.
- **Browser back button.** No effect on search state — search isn't a route.

## Unvalidated Decisions

These could reasonably go either way. Phi to confirm before sign-off, or
defer to prototype-validation.

- **Default printing = most recent.** Could also be cheapest, or
  most-popular-on-EDHrec (no API for the latter). Most recent matches what
  Thurgood is most likely to have just seen on EDHrec. Confirm.
- **Add button stays enabled with all warnings.** Strict reading of "Help,
  don't lecture" says yes. But for color identity violations in Commander
  specifically, there's an argument that the warning is so strong (the card
  literally cannot be played) that requiring a confirmation step is honest
  rather than nagging. Default position: enabled, soft warning. Confirm.
- **Sideboard view changes button label.** Could alternatively be a small
  "Main / Sideboard" segmented control inside the preview that defaults to
  whichever view is active but lets you redirect. Default position: button
  label change. Confirm if redirect is wanted.
- **Cmd-K shortcut on macOS in addition to `/`.** Common power-user pattern;
  zero cost to add. But adds a thing to maintain. Default position: include
  both. Confirm.
- **Preview collapse animation timing.** Spec says 150ms fade. Could be
  faster (snappy) or slower (calmer). Default 150ms; trust prototype on feel.

## Out of Scope

- **CardModal redesign for deck-context (art-swap-after-add).** Stays as
  today. Will be redesigned in a separate phase-1 spec, inheriting the
  inline-preview design language established here. Cross-check: no overlap —
  this spec touches only the search-add path, which currently routes through
  `CardModal` with `context='search'`. After this spec, that context is
  unused and the prop can be simplified or removed.
- **Brew mode / contextual suggestions.** Phase 3. Discovery-shaped features
  return there, in a different surface.
- **NLP parser deletion.** `nlpParser.ts` stays in the codebase, untouched.
  It may serve brew mode later. Search no longer routes through it; the
  parser is just unused for now.
- **Sidebar IA redesign.** Phase 2. After this spec, the sidebar has only a
  Decks tab. The collapsed rail still works. The lopsidedness is information
  for phase 2, not a problem to fix here. Cross-check: no overlap — we are
  removing a tab, not redesigning the sidebar's structure.
- **Mobile-responsive search.** Phase 2 (mobile is part of IA work). The
  inline-preview pattern is designed to scale better than the modal does, but
  the actual mobile breakpoints and layout are out of scope for this spec.
  Cross-check: prototype shows desktop layout only; mobile is a phase-2
  concern.
- **Capture-from-URL drag-and-drop path.** Phase 1, separate spec. The
  existing `DropOverlay` and capture-from-TCGPlayer-URL flow are untouched
  by this spec. Cross-check: no overlap — capture is an external→app path;
  search is a name-input→app path. Different entry points, both eventually
  feed `updateActiveDeck`.
- **Settings filter for "always show price in autocomplete" or similar.**
  Settings sprawl. The autocomplete is intentionally lean. If demand surfaces
  later, reconsider then.
